jquery trim

今天碰到个很诡异的问题，查了半天，原来是jquery的trim函数引起的

问题是这样的，之前的权限管理是我做的，现在需要修改权限管理的页面，将用户选择改为下拉并且分页，另一个同事负责修改，改完以后发现无法给用户的角色绑定数据了。
我去排查问题，

  1. 通过chrome的网络请求，看到用户查询角色的接口传递的参数不对，之前传递的是用户编码，现在传递的是用户名，这个很好发现，因为中文做了编码转换。
  2. 改为用户编码之后，发现这个接口还是查询不到用户对应的角色，我开始通过mongovue中直接去查看，发现也没有。还好数据量不多，一个个去对比，发现用户编码HQ01U4321该用户是存在的，那怎么查询不到了，这里折腾了好一会
  3. 我开始怀疑数据的问题了，结果突然别人发现HQ01U4321前面多了一个空格，这个空格怎么会传进去了，开始找数据的源头，这个用户名的数据来自ERP，去oracle中查看确实有一个空格，因为现在用的是测试库，所以数据有些问题是很正常的
  4. 但是就奇怪了，就算有空格也不会影响查询的结果，难道是url对空格做了处理……
  5. 过了好久之后，同事一看他的代码，他在取用户名的时候加了个trim函数，把空格去掉了，狂晕

总结：其实数据中包含空格是没有问题的，一般在数据的最源头可以处理，比如用户名，在用户注册的页面可以加上trim，这样存入数据库就没有问题了。如果原始数据就是有空格，也没有关系，因为url查询时对空格做了处理，为%20，所以在中间环节取值的时候不要用trim就可以避免问题的出现了。

2015-12-11

data transfer

最近需要上线一个项目，上线前的准备工作就是环境的搭建和数据迁移。数据迁移由我负责，主要工作就是将oracle数据库中的1亿多条数据导入到mysql和mongodb中。mysql中存放库存数据，mongodb中存放库存操作流水，通过一周的时间研究，得出如下结果

  1. mysql的innodb数据引擎能够存储的下1亿多条数据
  2. 为了减轻迁移的数据量，需要对迁移的数据进行处理，我这次处理的是库存数据，通过和业务确认，库存为0的数据可以不用初始化，一下子数据量较少到5000多万，同时将需要插入的sql字段确认，不必要的字段不处理
  3. 为了避免实时库存的影响，先将oralce中的数据导入到mysql，5000多万数据的导入时间在1个多小时左右，工具使用navicate，注意navicate的版本为Navicat Premium，这个版本及支持mysql，也支持oracle，记得设置oci.dll
  4. 语言的选择，由于我们的库存数据需要分表，分表依据为库存地code的MD5后hash取余数，所以直接使用sql实现难度较大，我选择通过编程实现，几种语言的对比在后面列出
  5. mongodb的插入速度实在惊人，把jvm的参数设置大一些后，每个线程可以插入20w,也就花费60秒左右

4.1java:可以很轻松实现MD5以及hash取余，使用多线程操作（ExecutorService+Runnable），数据库操作使用jdbctemplate或mybatis，发现jdbctemplate性能稍微好些

4.2python:可以很轻松实现MD5以及hash取余,使用executemany，但是executemany无法传入表名，因为我的表名是不确定的，如果表名确定的话，python的速度绝对超过java，同样插入5w条数据，python只要5秒左右，java需要50秒左右

4.3nodejs:看起来快，实际并不快，nodejs是异步操作，提交插入请求后很快返回结果，但是插入动作还在慢慢的执行，5w条数据，nodejs需要200多秒

最终数据顺利导入完成！选择了java+jdbctemplate

mysql deadlock index

mysql的死锁问题估计数据量稍微大了都有可能出现，例如经常出现的问题Caused by: com.mysql.jdbc.exceptions.jdbc4.MySQLTransactionRollbackException: Deadlock found when trying to get lock; try restarting transaction。很熟悉吧，今天项目中就出现了这个问题，我操作的是库存表，估计搞库存的都会可能碰到库存加减并发的问题，在此不去讨论库存的具体业务逻辑，而是针对mysql的死锁问题做一个总结。

mysql有行锁和表锁区分。在navicate客户端开启两个查询窗口即可模拟操作。下面的测试是以mysql的innodb数据引擎为基础。

1.测试行锁

命令行1
begin；
select * from user where id = 1 for update；

命令行2
（1）update user set username = “test123” where id = 1;
（2）update user set username = “test123” where id = 2;

命令行1没有commit之前，
命令行2中的第一条命令无法更新成功，命令行2中的第二条命令可以更新成功

2.测试表锁

命令行1
begin；
select * from user where username = “test” for update;

命令行2
（1）update user set username = “test123” where id = 1;
（2）update user set username = “test123” where id = 2;
（3）update user set username = “test123” where username = “test”;

命令1没有commit之前，
命令行2中的三条命令无法都更新成功

3.如果既要行锁但是又要查询条件并不是主键，可以使用如下方法（子查询）
begin；
select * from user where id = (select id fro user where username = “test”) for update;

要保证子查询select的结果唯一，如果结果不唯一，则报如下错误
mysql> select * from user where id = (select id from user where username = “test”) for update;
1242 - Subquery returns more than 1 row

这时候可以使用in，不过锁住的是整个表
begin;
select * from user where id in (select id from user where username = “test”) for update;

同时还有另外一种方法，就是给where 后面的条件加上唯一索引（或者普通索引，但是锁住的可能是多行，mysql查询结果总量大于等于3就不行了，所以最好用唯一索引），如下：

命令行1
begin;
select * from inventory_0 where goods_code = ‘001’ and ipwareh_code = ‘HQ0001’ and acwareh_code = ‘HQ0001’ for update;

命令行2
update inventory_0 set inventory_quantity = 13 where id = 1;

命令行3
update inventory_0 set inventory_quantity = 13 where goods_code = ‘001’ and ipwareh_code = ‘HQ0001’ and acwareh_code = ‘HQ0001’;

这时候命令行1没哟提交之前，命令行2，命令行3不能执行

4.默认的update语句也是区分锁表和锁行
update user set username = “test1” where id = 1;则锁行,id为主键
update user set username = “test1” where username = “test”;则锁表，username 没有索引或者只是普通索引

first be interviewer 

工作了4年，今天第一次当面试官，从今天开始，打算将一些事情记录下来。

原因很简单，第一次当面试官很不成功。

今天在写程序，突然被老大叫去面试一个应届生。顿时感觉手忙脚乱，人事递给我简历，我立马回答说
我先上个厕所。回来的时候，我拿着简历，顺便也拿了手机，想想还是别带手机了。就这样，手机摔到 地上了，还好，没摔坏。
在路上边走边看，到了面试的房间，看到了那个应届生。按照自己的面试经验，先让他做个自我简介。
什么，94年的，瞬间感觉自己老了。
上海某个二本学校的。其实这个学校感觉还可以，因为自己当时差点报了这个学校。
接下来是我们的对话

Q：大学学了些什么专业课？
A：C语言，操作系统原理，这学期正在学。
Q：没有其他课了？
A：记不得了……
Q：数据结构、组成原理、计算机网络、数据库、编译原理……我一下子说了好多，你有自己动手写过程序吗？
A:没有
Q：一点都没写过？
A：我报名了一个java的培训班
Q：那好啊，给我说说学了些什么？
A：学了SSH框架
Q：自己有尝试搭建过SSH吗?
A: 没有
Q：那知道这3个框架都分别干什么吗？
A：不太清楚，当时学的时候还知道，现在忘记了
Q：其实忘记很正常，尤其是在学校没做过项目，很多知识学了就忘，java你听说过吧？
A：听说过，自己还学了点，不过也没写过什么
Q:那你把知道说出来，知道多少说多少
A：没太写过，感觉都忘了
Q：我感觉不知道问什么了，我就看着简历后面的面试题，看了他哪些不会做，接下来就是给他讲解题目
A：聆听中………………
Q：希望这些知识回去自己看看，说不定下家面试还能用得上，你先等一会

我从屋子里走出来了，和老大说了说，其他人都过来问怎么样，怎么样

结果很遗憾的就是让人事和他说bye，bye了

回去我感觉挺不好意思的，好几天感觉都挺内疚，不知道自己是不是问的太多了，还是……

我觉得我好像不太适合当面试官
