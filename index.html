travel chengdu

最近休假去成都呆了一个多星期，感觉和上海确实有很多不同。

1.成都的节奏相对比较慢，街上或者公园里经常看到下棋的、打麻将的、喝茶的、跳舞的……
2.成都的交通拥挤，而且周一到周五还有车辆限行，周一限行车牌尾号为1和6、周二限行车牌尾号为2和7，……周五限行车牌尾号为5和0，还有部分路段在某些时间段是单行，其他时间段双向行驶，这点刚来成都自驾的要注意。
3.成都的夜生活很丰富，我第一天晚上去吃饭喝酒，一直吃到夜里12点多，11点多的时候还有很多人下来买点酒和菜拎回家。
4.成都人很喜欢运动，看到很多人打篮球和乒乓球，而且乒乓球还有比赛，我还看到了有人抽打陀螺，自己体验了下，感觉很有意思。
5.当地方言比较明显，出去玩基本都说出去耍，还有什么要得、要得。
6.成都麻将很有意思，街上到处都是机麻（麻将），谁放炮谁给钱、自抠三门都要给、杠还有区分是不是小雨之类的，麻将挺有意思。
7.成都的房子特别多，房价总体还是能接受，房租不算太贵，吃喝成本相对当地人来说也不算低，地铁挺方便，公交也不错，有公交专用车道。
8.成都的高校挺多，去了下西南交大和川大，学校环境还不错，但是有个地方相对上海比较奇怪，大学里有自己的附属幼儿园和附属小学，比如西安交大附属的子弟学校。
9.成都的妹子比较多，尤其是市中心和地铁上，感觉女的比男的多一些，只是感觉啊，没有统计过……
10.小吃很多，酸辣粉、龙抄手、烤秋刀鱼、兔子头（这个我没敢吃）、糖油果子（街上到处都是，上海的很喜欢）、冒菜、火锅，这些在宽窄巷子和锦里到处都是
11.成都给人感觉很安全，到处都是警察，大的路口都有交警，地铁上偶尔也有巡逻的，问路很方便

还有很多，一时半会想不出来了，总之，去了之后感觉很舒服，希望大家有空都去体验下，尤其是一线城市忙碌的coder

kafka demo

最近公司有个实时日志分析，准备采用kafka，自己在windows机器上尝试搭建了一下，下面是搭建步骤
1.zookeeper集群为了方便就一台机器 localhost:2181
2.下载kafka，记得下载带scala的预编译版本
https://www.apache.org/dyn/closer.cgi?path=/kafka/0.9.0.1/kafka_2.11-0.9.0.1.tgz
3.解压，并重命名为多个文件夹
4.修改配置文件，目前只需要需改的地方有3个，假设kafka集群机器有两台机器，则server.properties需要修改如下3个地方
（1）broker.id=0
listeners=PLAINTEXT://:9092
log.dirs=/tmp0/kafka-logs
（2）broker.id=1
listeners=PLAINTEXT://:9093
log.dirs=/tmp1/kafka-logs
5.启动，先启动zookeeper，后启动两台kafka
kafka的启动需要到bin/windows目录下，运行如下命令
kafka-server-start.bat ../../config/server.properties
6.测试，同样需要进入bin/windows目录下，然后运行命令
（1）创建topic
kafka-topics.bat -zookeeper localhost:2181 -topic test -replication-factor 2 -partitions 5 -create
（2）查看topic
kafka-topics.bat -zookeeper localhost:2181 -list
（3）创建producer
kafka-console-producer.bat -broker-list localhost:9092,localhost:9093 -topic test
（4）创建consumer
kafka-console-consumer.bat -zookeeper localhost:2181 - from-begining -topic test
（5）发送消息

java concurrent lock

Java中的线程同步机制主要有以下几种方式
1.volatile变量：轻量级多线程同步机制，不会引起上下文切换和线程调度。仅提供内存可见性保证，不提供原子性。
2.CAS原子指令：轻量级多线程同步机制，不会引起上下文切换和线程调度。它同时提供内存可见性和原子化更新保证。
3.内部锁和显式锁：重量级多线程同步机制，可能会引起上下文切换和线程调度，它同时提供内存可见性和原子性。

Java中锁类型如下
1.Synchronized锁：内部锁，锁的获取和释放不需要自己手动写程序代码，底层的实现是监控Monitor对象，通过MonitorEnter和MonitorExist方式实现。该锁是重量级锁，一个线程获得该锁后，其他所有线程都要等待。
2.Lock锁：显示锁，锁的获取和释放需要自己手动写程序代码，一般如下，内部实现通过Java的AQS实现，而AQS的实现依赖于CAS。该锁是轻量级锁，是乐观锁，只有两个线程同时写才会让另一个线程等待，读不需要等待。

Lock锁相对Synchronized锁有如下好处
1.Lock可以按需要加锁与释放锁，便于手动管理
2.Lock可以设置超时时间
3.Lock锁可以响应中断
4.Lock锁一般是乐观锁，Synchronized锁一般是悲观锁

现代的多处理器系统大多提供了特殊的指令来管理对共享数据的并发访问，这些指令能实现原子化的读-改-写操作。现代典型的多处理器系统通常支持两种同步原语（机器级别的原子指令）：CAS 和 LL/SC。
1.Intel，AMD 和 SPARC 的多处理器系统支持“比较并交换”（compare-and-swap，CAS）指令。
2.IBM PowerPC，Alpha AXP，MISP 和 ARM 的多处理器系统支持“加载链接 / 存储条件”（load-linked/store-conditional，LL/SC）指令。
3.CAS实际上是利用处理器提供的CMPXCHG指令实现的，而处理器执行CMPXCHG指令是一个原子性操作。

redis queue

互联网公司对redis的喜欢毋庸置疑，redis能实现的功能太多，不用业务有不同的方法。对于很多业务，redis总感觉能帮得上忙。最近买了本redis的新书，看着看着，想起来之前一个项目中使用过redis，而且效果很好。

首先，总体回顾下redis的5种数据类型

1.String
2.List
3.Hash
4.Set
5.Sorted Set

接着，简单介绍下业务。我是做电商开发的，电商的业务比较复杂，最头疼的问题有，库存问题和抢购问题。我之前负责的促销系统和优惠券系统开发，这里面redis的身影到处都是，抢红包就是给我印象很深刻的一次。当时是现金红包，抢到之后没有门槛的限制，所以每次都是人数爆满。有几次导致整个系统差点崩溃，后来用了redis模拟队列的方式解决了主要问题。当然其他系统也做了相应的修改，比如，用户登录接口的修改，微信的校验，等等。现在主要来说说redis的用法。

队列的方法有很多，为什么非要使用redis的方式来实现，理由大概有如下几点

1.java的消息队列，如activemq或者淘宝的metaq等等，主要的缺点是每秒钟能处理的消息数量太小（每秒千级别），和redis相比（每秒万级别）还是有差距的，而且二者的使用对象也不太相同，采用redis后，用户抢到现金红包后可以立即到账户查看，而且这个是一般用户希望的结果
2.java自身的队列，如LinkedList（一般队列）和ArrayBlockingQueue（阻塞队列），这个也是在内存中处理的，但是无法实现消息的持久化和程序解耦，redis的持久化功能就不用说了，还有redis能够方便的在不同程序间使用

选择redis的理由已经充分，但是redis中没有队列的数据类型，所以需要自己实现。实现的方式很简单，就是使用redis的List数据类型，实现java的Queue接口即可，将需要的方法实现。下面给出代码

jquery trim

今天碰到个很诡异的问题，查了半天，原来是jquery的trim函数引起的

问题是这样的，之前的权限管理是我做的，现在需要修改权限管理的页面，将用户选择改为下拉并且分页，另一个同事负责修改，改完以后发现无法给用户的角色绑定数据了。
我去排查问题，

  1. 通过chrome的网络请求，看到用户查询角色的接口传递的参数不对，之前传递的是用户编码，现在传递的是用户名，这个很好发现，因为中文做了编码转换。
  2. 改为用户编码之后，发现这个接口还是查询不到用户对应的角色，我开始通过mongovue中直接去查看，发现也没有。还好数据量不多，一个个去对比，发现用户编码HQ01U4321该用户是存在的，那怎么查询不到了，这里折腾了好一会
  3. 我开始怀疑数据的问题了，结果突然别人发现HQ01U4321前面多了一个空格，这个空格怎么会传进去了，开始找数据的源头，这个用户名的数据来自ERP，去oracle中查看确实有一个空格，因为现在用的是测试库，所以数据有些问题是很正常的
  4. 但是就奇怪了，就算有空格也不会影响查询的结果，难道是url对空格做了处理……
  5. 过了好久之后，同事一看他的代码，他在取用户名的时候加了个trim函数，把空格去掉了，狂晕

总结：其实数据中包含空格是没有问题的，一般在数据的最源头可以处理，比如用户名，在用户注册的页面可以加上trim，这样存入数据库就没有问题了。如果原始数据就是有空格，也没有关系，因为url查询时对空格做了处理，为%20，所以在中间环节取值的时候不要用trim就可以避免问题的出现了。

data transfer

最近需要上线一个项目，上线前的准备工作就是环境的搭建和数据迁移。数据迁移由我负责，主要工作就是将oracle数据库中的1亿多条数据导入到mysql和mongodb中。mysql中存放库存数据，mongodb中存放库存操作流水，通过一周的时间研究，得出如下结果

  1. mysql的innodb数据引擎能够存储的下1亿多条数据
  2. 为了减轻迁移的数据量，需要对迁移的数据进行处理，我这次处理的是库存数据，通过和业务确认，库存为0的数据可以不用初始化，一下子数据量较少到5000多万，同时将需要插入的sql字段确认，不必要的字段不处理
  3. 为了避免实时库存的影响，先将oralce中的数据导入到mysql，5000多万数据的导入时间在1个多小时左右，工具使用navicate，注意navicate的版本为Navicat Premium，这个版本及支持mysql，也支持oracle，记得设置oci.dll
  4. 语言的选择，由于我们的库存数据需要分表，分表依据为库存地code的MD5后hash取余数，所以直接使用sql实现难度较大，我选择通过编程实现，几种语言的对比在后面列出
  5. mongodb的插入速度实在惊人，把jvm的参数设置大一些后，每个线程可以插入20w,也就花费60秒左右

4.1java:可以很轻松实现MD5以及hash取余，使用多线程操作（ExecutorService+Runnable），数据库操作使用jdbctemplate或mybatis，发现jdbctemplate性能稍微好些

4.2python:可以很轻松实现MD5以及hash取余,使用executemany，但是executemany无法传入表名，因为我的表名是不确定的，如果表名确定的话，python的速度绝对超过java，同样插入5w条数据，python只要5秒左右，java需要50秒左右

4.3nodejs:看起来快，实际并不快，nodejs是异步操作，提交插入请求后很快返回结果，但是插入动作还在慢慢的执行，5w条数据，nodejs需要200多秒

最终数据顺利导入完成！选择了java+jdbctemplate
