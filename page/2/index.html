mysql deadlock index

mysql的死锁问题估计数据量稍微大了都有可能出现，例如经常出现的问题Caused by: com.mysql.jdbc.exceptions.jdbc4.MySQLTransactionRollbackException: Deadlock found when trying to get lock; try restarting transaction。很熟悉吧，今天项目中就出现了这个问题，我操作的是库存表，估计搞库存的都会可能碰到库存加减并发的问题，在此不去讨论库存的具体业务逻辑，而是针对mysql的死锁问题做一个总结。

mysql有行锁和表锁区分。在navicate客户端开启两个查询窗口即可模拟操作。下面的测试是以mysql的innodb数据引擎为基础。

1.测试行锁

命令行1
begin；
select * from user where id = 1 for update；

命令行2
（1）update user set username = “test123” where id = 1;
（2）update user set username = “test123” where id = 2;

命令行1没有commit之前，
命令行2中的第一条命令无法更新成功，命令行2中的第二条命令可以更新成功

2.测试表锁

命令行1
begin；
select * from user where username = “test” for update;

命令行2
（1）update user set username = “test123” where id = 1;
（2）update user set username = “test123” where id = 2;
（3）update user set username = “test123” where username = “test”;

命令1没有commit之前，
命令行2中的三条命令无法都更新成功

3.如果既要行锁但是又要查询条件并不是主键，可以使用如下方法（子查询）
begin；
select * from user where id = (select id fro user where username = “test”) for update;

要保证子查询select的结果唯一，如果结果不唯一，则报如下错误
mysql> select * from user where id = (select id from user where username = “test”) for update;
1242 - Subquery returns more than 1 row

这时候可以使用in，不过锁住的是整个表
begin;
select * from user where id in (select id from user where username = “test”) for update;

同时还有另外一种方法，就是给where 后面的条件加上唯一索引（或者普通索引，但是锁住的可能是多行，mysql查询结果总量大于等于3就不行了，所以最好用唯一索引），如下：

命令行1
begin;
select * from inventory_0 where goods_code = ‘001’ and ipwareh_code = ‘HQ0001’ and acwareh_code = ‘HQ0001’ for update;

命令行2
update inventory_0 set inventory_quantity = 13 where id = 1;

命令行3
update inventory_0 set inventory_quantity = 13 where goods_code = ‘001’ and ipwareh_code = ‘HQ0001’ and acwareh_code = ‘HQ0001’;

这时候命令行1没哟提交之前，命令行2，命令行3不能执行

4.默认的update语句也是区分锁表和锁行
update user set username = “test1” where id = 1;则锁行,id为主键
update user set username = “test1” where username = “test”;则锁表，username 没有索引或者只是普通索引

first be interviewer 

工作了4年，今天第一次当面试官，从今天开始，打算将一些事情记录下来。

原因很简单，第一次当面试官很不成功。

今天在写程序，突然被老大叫去面试一个应届生。顿时感觉手忙脚乱，人事递给我简历，我立马回答说
我先上个厕所。回来的时候，我拿着简历，顺便也拿了手机，想想还是别带手机了。就这样，手机摔到 地上了，还好，没摔坏。
在路上边走边看，到了面试的房间，看到了那个应届生。按照自己的面试经验，先让他做个自我简介。
什么，94年的，瞬间感觉自己老了。
上海某个二本学校的。其实这个学校感觉还可以，因为自己当时差点报了这个学校。
接下来是我们的对话

Q：大学学了些什么专业课？
A：C语言，操作系统原理，这学期正在学。
Q：没有其他课了？
A：记不得了……
Q：数据结构、组成原理、计算机网络、数据库、编译原理……我一下子说了好多，你有自己动手写过程序吗？
A:没有
Q：一点都没写过？
A：我报名了一个java的培训班
Q：那好啊，给我说说学了些什么？
A：学了SSH框架
Q：自己有尝试搭建过SSH吗?
A: 没有
Q：那知道这3个框架都分别干什么吗？
A：不太清楚，当时学的时候还知道，现在忘记了
Q：其实忘记很正常，尤其是在学校没做过项目，很多知识学了就忘，java你听说过吧？
A：听说过，自己还学了点，不过也没写过什么
Q:那你把知道说出来，知道多少说多少
A：没太写过，感觉都忘了
Q：我感觉不知道问什么了，我就看着简历后面的面试题，看了他哪些不会做，接下来就是给他讲解题目
A：聆听中………………
Q：希望这些知识回去自己看看，说不定下家面试还能用得上，你先等一会

我从屋子里走出来了，和老大说了说，其他人都过来问怎么样，怎么样

结果很遗憾的就是让人事和他说bye，bye了

回去我感觉挺不好意思的，好几天感觉都挺内疚，不知道自己是不是问的太多了，还是……

我觉得我好像不太适合当面试官

java jvm gc

垃圾回收（GC，Garbage Collection）的用途，释放没有被引用的对象，整理内存碎片。

判定对象为垃圾对象的方法
1.引用计数法（Reference Counting），无法解决循环依赖的情况，Java没有用该方法，Python使用该方法
2.跟踪法/可达性分析法（Tracing），查找根节点到对象的可达（reachable）路径确定是否删除

常用垃圾回收算法
1.标记-清除（Mark-Sweep）算法
2.标记-整理/压缩（Mark-Compact）算法
3.复制（Copying）算法
4.分代收集（Generational Collection）算法

JDK1.7中的具体垃圾收集器，垃圾回收算法的具体实现
1.Serial
2.Serial Old
3.ParNew
4.Parallel Scavenge
5.Parallel Old
6.CMS(Concurrent-Mark-Sweep)
7.G1

Java中的垃圾回收包含Minor GC，Major GC，Full GC
1.Minor GC发生在年轻代，发生比较频繁，速度较快，采用Copying算法
2.Major GC发生在年老代，速度较慢，时间一般是Minor GC的10倍，采用Mark-Compact算法
3.Full GC不等于Major GC，也不等于Minor GC + Major GC，有时候确实等于Major GC，但也有时候等于Minor GC + Major GC，具体是可以通过参数进行配置，可以通过-XX:+PrintGC 输出GC日志以及-XX:+PrintGCDetails 输出GC的详细日志

java jvm

Java内存（逻辑内存）区域划分
1.堆（Heap）
2.虚拟机栈（Stack），用于存储局部变量表、操作栈、动态链接、方法出口等信息
3.方法区（Method Area），用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据
4.本地方法栈（Native Method Stack），用于存储Native方法需要的信息
5.程序计数器（PC，Program Counter Register），这个计数器的值来选取下一条需要执行的字节码指令，顺序、分支、循环、跳转、异常处理、线程恢复等基础功能都需要依赖这个计数器来完成。
6.运行时常量池(Runtime Constant Pool)，用于存放编译期生成的各种字面量和符号引用，String 类的intern() 方法

Java堆的分区
1.年轻代/新生代（Young Generation），又分为伊甸（Eden）、幸存区0（Survivor From）、幸存区1（Survivor To），三个区域的默认比例为8:1:1，可以通过-XX:SurvivorRatio设置。可以通过-XX:NewSize（for jdk 1.3/1.4）或-Xmn(for jdk 1.4 or later)设置整个年轻代大小
2.年老代（Old Generation），年老代与年轻代的默认比例为2:1，可以通过-XX:NewRatio设置
3.永久代（Permanent Generation），Hot Spot虚拟机中方法区近似于永久代，BEA JRockit、IBM J9等来说是不存在永久代的概念的，可以通过-XX:PermSize与-XX:MaxPermSize设置初始值和最大值

年轻代和年老代的所有区域大小可以通过-Xms与-Xmx设置初始值和最大值，两者最好设置相同，否则capacity的值会来回飘动，避免每次垃圾回收完成后JVM重新分配内存

-Xms 默认大小为物理内存的1/64
-Xmx 默认大小为物理内存的1/4，最大值与操作系统的数据模型有关、虚拟内存、物理内存有关
-XX:PermSize 我的64为机器为20M左右,如果系统中反射应用比较多，可以将这项设置大一些
-XX:MaxPermSize 32位机器默认的永久代的大小为64M，64位的机器则为85M，我的64为机器为82M左右
-Xss 每个线程堆栈的大小，JDK5.0以后每个线程堆栈大小为1M,以前每个线程堆栈大小为256K

-X选项是非标准选项，如有更改，恕不另行通知。
可以通过java -X查看
